# 5.1 Actuación 1: Plataforma de Ciudad Inteligente (PCI) - Mantenimiento

## 5.1.1 Objeto del manual de mantenimiento

La presente sección describe los procedimientos de mantenimiento preventivo y correctivo de
la Plataforma de Ciudad Inteligente, incluyendo actualizaciones de software, monitorización
de rendimiento y gestión de incidencias técnicas.

## 5.1.2 Equipamiento y componentes sujetos a mantenimiento

| Componente | Tipo de mantenimiento | Frecuencia |
| --- | --- | --- |
| Servidor de aplicación | Software, parches de seguridad | Mensual |
| Base de datos | Integridad, backup, optimización | Semanal |
| Motor de búsqueda | Reindexación, limpieza de caché | Quincenal |
| Integración de datos | Validación de conexiones | Semanal |
| Certificados SSL | Renovación, validación | Trimestral |

## 5.1.3 Plan de mantenimiento preventivo

| Tarea | Descripción | Frecuencia | Responsable |
| --- | --- | --- | --- |
| Monitorización de salud | Revisar métricas de CPU, memoria y disco | Diaria | Equipo técnico |
| Backup de base de datos | Crear copia de seguridad completa | Diaria | Automatizado + validación |
| Limpieza de logs | Archiver logs antiguos y liberar espacio | Semanal | Script automatizado |
| Actualización de parches | Instalar parches de seguridad críticos | Mensual | Administrador sistemas |
| Validación de integridad | Verificar consistencia de datos | Quincenal | DBA |

## 5.1.4 Procedimiento de actualización de software

| Paso | Instrucción |
| --- | --- |
| 1 | Revisar notas de release de nueva versión. |
| 2 | Realizar backup completo en entorno productivo. |
| 3 | Aplicar actualización en entorno de prueba primero. |
| 4 | Validar funcionalidades críticas en entorno de prueba. |
| 5 | Programar ventana de mantenimiento en horario de baja demanda. |
| 6 | Ejecutar actualización en producción. |
| 7 | Validar integraciones y funcionalidad post-actualización. |
| 8 | Documentar cambios y versión resultante. |

## 5.1.5 Gestión de incidencias técnicas

| Nivel de gravedad | Descripción | Tiempo respuesta | Tiempo resolución |
| --- | --- | --- | --- |
| Crítico | Sistema no disponible | 1 hora | 3 horas |
| Alto | Funcionalidad degradada | 2 horas | 8 horas |
| Medio | Funcionalidad parcialmente afectada | 4 horas | 24 horas |
| Bajo | Inconveniente sin impacto operativo | 8 horas | 1 semana |

## 5.1.6 Procedimiento de respuesta a incidencias

| Paso | Instrucción |
| --- | --- |
| 1 | Registrar incidencia con descripción detallada y hora de inicio. |
| 2 | Asignar nivel de gravedad según impacto operativo. |
| 3 | Ejecutar diagnóstico inicial para identificar causa. |
| 4 | Implementar solución temporal si es aplicable. |
| 5 | Resolver causa raíz del problema. |
| 6 | Validar resolución con usuario de origen o administrador. |
| 7 | Documentar lección aprendida y mejora preventiva. |

## 5.1.7 Monitorización de rendimiento

| Métrica | Umbral alerta | Acción recomendada |
| --- | --- | --- |
| CPU promedio | >80% | Optimizar consultas o escalar infraestructura |
| Memoria utilizada | >85% | Revisar procesos y ajustar asignación |
| Espacio en disco | >80% | Archivar datos históricos o expandir almacenamiento |
| Tiempo respuesta API | >2 segundos | Analizar consultas lentas y optimizar índices |
| Disponibilidad del servicio | <99.5% | Investigar causa raíz y aplicar corrección |

## 5.1.8 Procedimiento de backup y recuperación

| Acción | Descripción | Frecuencia |
| --- | --- | --- |
| Backup completo | Copia integral de base de datos y datos de aplicación | Diaria |
| Backup incremental | Copia de cambios desde último backup completo | Cada 4 horas |
| Verificación de backup | Validar integridad de archivo de backup | Semanal |
| Prueba de restauración | Simular recuperación en entorno aislado | Mensual |
| Archivo de backup | Mover backups antiguos a almacenamiento de largo plazo | Trimestral |

## 5.1.9 Documentación de cambios

| Elemento | Contenido | Actualización |
| --- | --- | --- |
| Registro de cambios | Versión, fecha, cambios realizados, responsable | Cada actualización |
| Matriz de compatibilidad | Versión de software vs. integraciones externas | Mensual |
| Runbook de recuperación | Procedimientos paso a paso para restauración | Trimestral |
| Configuración de sistema | Parámetros actuales y valores por defecto | Semestral |

## 5.1.10 Gestión de incidencias de mantenimiento

| Incidencia | Actuación recomendada |
| --- | --- |
| Actualización falla | Realizar rollback a versión anterior y diagnosticar error. |
| Degradación post-actualización | Monitorizar intensamente y aplicar parches si es necesario. |
| Backup fallido | Investigar causa y ejecutar backup manual inmediatamente. |
| Rendimiento crítico | Activar modo de degradación limitada si aplica. |

## 5.1.11 Buenas prácticas de mantenimiento

1. Mantener ventanas de mantenimiento coordinadas fuera de horarios críticos.
2. Comunicar con antelación cambios planificados a usuarios finales.
3. Documentar exhaustivamente cada intervención realizada.
4. Realizar pruebas de restauración regularmente (mínimo mensual).
5. Establecer monitorización 24/7 de sistemas críticos.
